# Portfolio Website Requirements v0.1

## Document Status / 文档状态

EN:
This document is a working requirements draft for the portfolio website. It defines the current project direction, execution boundaries, and early-stage priorities without locking the final website too early. It should be updated as the portfolio grows.

CN:
这是一份作品集网站的工作版需求文档，用来定义当前项目方向、执行边界和早期阶段的优先级，但不会过早把最终网站完全定死。后续随着作品集推进，这份文档应持续更新。

## Execution Rule / 执行规则

EN:
The English version is the main execution reference for Codex. The Chinese version explains my creative intention and helps me review the project direction. If the two versions conflict, follow the English version unless I explicitly say otherwise.

CN:
英文是 Codex 的主要执行依据；中文用于解释我的创作意图，也方便我自己检查项目方向。如果中英文内容有冲突，默认以英文为准，除非我特别说明。

## Website Goal / 网站目标

EN:
This website is being built for 2027 Fall Hong Kong master’s applications and should also function as one of the portfolio projects itself. It is not only a container for project pages, but a designed interface that demonstrates visual system thinking, digital presentation ability, and editorial curation. The website should support both application review and future portfolio-driven creative opportunities.

CN:
这个网站是为 2027 Fall 香港硕士申请而建立的，同时它本身也应当作为作品集项目之一存在。它不只是一个装作品页面的容器，而应该是一个被设计过的界面，用来体现视觉系统思维、数字呈现能力和编辑式策展能力。它既要服务申请场景，也要支持未来与作品集相关的创意职业机会。

## Target Viewers / 目标受众

EN:
The primary viewers are admissions reviewers, faculty, interviewers, and potential collaborators. Secondary viewers include creative recruiters, employers, and people quickly opening portfolio links on desktop or mobile. The website must work for both short scanning and deeper reading, meaning the overall structure should be immediately understandable while still supporting detailed project exploration.

CN:
主要受众包括招生审核者、项目老师、面试官和潜在合作方。次级受众包括创意行业招聘方、雇主，以及通过电脑或手机快速点开链接查看的人。网站必须同时适用于“快速浏览”和“深入阅读”两种场景，也就是说整体结构要一眼能懂，同时又能支持更深入的项目阅读。

## Core Message / 核心信息

EN:
The website should communicate the following core message: I build visual systems that turn identity, commerce, cultural memory, and interactive storytelling into applied creative experiences. The portfolio should show that the work is not only image-making, but also audience-aware, process-aware, commercially aware, and structurally considered.

CN:
网站需要传达的核心信息是：我构建视觉系统，把身份、商业、文化记忆和互动叙事转化为可被应用的创意体验。作品集应当让人看到，我做的不只是图像本身，而是同时考虑受众、过程、商业应用和结构逻辑的创作。

## Non-negotiables / 不可妥协原则

EN:
All final portfolio content must be based on original work, original photography, original design, original process documents, and real project materials. The website must not rely on stock imagery, copied designs, unlicensed assets, invented clients, invented outcomes, fake case studies, or generic AI-generated final artworks used as substitutes for the owner’s authorship. AI may support research, organization, prototyping, coding, and workflow documentation, but it must not replace authorship in the final portfolio outcomes. The website should prioritize clarity, originality, process visibility, honest scope, and careful curation over filler or visual noise.

CN:
所有最终作品集内容都必须基于原创作品、原创摄影、原创设计、原创过程文档和真实项目材料。网站不能依赖素材图库图片、抄袭设计、未经授权的资产、虚构客户、虚构结果、伪造案例，或用泛化的 AI 成图来代替作者本人的创作。AI 可以用于研究、整理、原型、编程和流程文档支持，但不能替代最终作品的作者性。网站应优先强调清晰度、原创性、过程可见性、真实范围和谨慎筛选，而不是用凑数内容或视觉噪音填满页面。

## Main Pages / 主要页面

EN:
The early website should include a homepage, a projects index, four main project pages, an about page, and a custom 404 page. These pages should provide a stable and readable first version of the portfolio. The homepage should introduce the overall direction. The projects index should present the main case studies as a structured archive rather than a random gallery. The about page should explain positioning, and the custom 404 page should still feel consistent with the archive/interface concept.

CN:
网站早期版本应包含：首页、项目总览页、四个主要项目页、About 页面和自定义 404 页面。这些页面要组成一个稳定且可阅读的第一版作品集。首页负责建立整体方向，项目总览页应该像一个结构化档案入口，而不是随意堆图的画廊；About 页面负责说明定位；404 页面也应延续档案/界面这一概念，而不是成为脱离整体风格的普通报错页。

## Main Projects / 主要项目

EN:
The website should initially center on four main projects. The Interactive Portfolio Website project treats the website itself as a case study and should eventually include sitemap, wireframes, layout evolution, screenshots, mobile adaptation, and navigation logic. The 404 Self Live2D Interactive Avatar project should translate the user’s original fragmented identity visual language into an interactive character system through sketches, redesign, layer breakdown, rigging, expression states, and a final demo. The AI-assisted E-commerce Brand System project should show a real applied workflow from research and insight to branding, packaging, content, and AI-assisted support. The Portrait Persona System project should show portrait photography as identity construction through persona grouping, contact sheets, editing logic, lighting or styling decisions, and final image selection.

CN:
网站初期应围绕四个主要项目展开。Interactive Portfolio Website 项目把网站本身作为案例研究，后续应逐步展示 sitemap、wireframe、版式演化、截图、移动端适配和导航逻辑。404 Self Live2D Interactive Avatar 项目需要把你原本关于碎片身份的视觉语言转化为一个互动角色系统，通过草图、重设计、分层、绑定、表情状态和最终演示来呈现。AI-assisted E-commerce Brand System 项目要展示一个真实的应用型流程，从研究和洞察一直到品牌、包装、内容和 AI 辅助工作流。Portrait Persona System 项目则要把人像摄影呈现为“身份构建”，通过 persona 分组、contact sheet、修图逻辑、灯光或造型决策以及最终选片来说明。

## Project Page Structure / 项目页面结构

EN:
Each main project page should follow a reusable case-study structure so the portfolio remains coherent even as content grows. The default structure should include: hero, one-line concept, context, objective, role, process, final outputs, and reflection. This structure should support both quick understanding and deeper reading. It should also leave enough room for project-specific variations without collapsing into a generic template.

CN:
每个主要项目页面都应遵循可复用的案例研究结构，这样即使内容不断增加，整个作品集仍然保持一致性。默认结构应包含：hero、one-line concept、context、objective、role、process、final outputs 和 reflection。这个结构既要支持快速理解，也要支持更深入阅读。同时它也要保留足够空间，让不同项目能够有自己的差异，而不是被压成千篇一律的模板。

## Process Materials to Support / 需要支持的过程材料

EN:
The website must support process documentation as a core layer rather than an afterthought. Depending on the project, process materials may include sketches, contact sheets, wireframes, screenshots, before-and-after comparisons, workflow diagrams, prompt screenshots, packaging drafts, research boards, rigging screenshots, layer breakdowns, version captures, and selection logic. Process materials should help explain decisions, not simply increase volume.

CN:
网站必须把过程文档作为核心层来支持，而不是事后补充。根据项目不同，过程材料可能包括草图、contact sheet、wireframe、截图、前后对比、工作流图、prompt 截图、包装草稿、研究板、绑定截图、分层说明、版本记录和筛选逻辑等。这些过程材料的作用是帮助解释决策，而不是单纯增加页面数量。

## Visual Direction / 视觉方向

EN:
The website should feel like a quiet visual archive and interactive interface, not a generic startup, SaaS, or corporate portfolio template. It should be editorial, image-forward, intentional, process-friendly, and slightly experimental without losing clarity. Abstract style words must be interpreted with boundaries. Dreamcore here should mean quiet, empty, memory-like, soft, and slightly unreal; it must not become horror, heavy cyberpunk, excessive neon, or random surreal decoration. Glitch should be subtle, restrained, and narrative-driven; it should suggest fragmentation, digital residue, or unstable memory without becoming visually chaotic. Archive-like should mean calm structure, room for sequencing, and attention to documentation, not dusty nostalgia for its own sake. The visual language should develop from real project content and should not depend on trend-driven styling detached from the work itself.

CN:
网站应当像一个安静的视觉档案库和互动界面，而不是普通的 startup、SaaS 或 corporate portfolio 模板。整体应偏 editorial、以图像为主、有明确意图、适合放过程材料，并且带有一点实验性，但不能失去清晰度。那些抽象风格词必须有明确边界。这里的 Dreamcore 应理解为安静、空旷、像记忆、柔和、略带不真实感；不能做成恐怖风、重赛博朋克、过量霓虹，或随意拼贴式的超现实装饰。Glitch 应该是克制的、轻微的、由叙事驱动的；它可以暗示碎片化、数字残留或不稳定记忆，但不能变成视觉混乱。Archive-like 的意思是结构平静、允许序列化阅读、重视文档痕迹，而不是为了“复古档案感”去做无意义的旧化。整体视觉语言应从真实项目内容生长出来，而不是脱离作品本身去追逐流行风格。

## Interaction Direction / 交互方向

EN:
Interaction should support reading, sequencing, and project atmosphere rather than competing with the work. Early interactions should remain stable, readable, and low-complexity. Navigation should be easy to understand without requiring experimentation to access content. Motion, transitions, and interactive details may later reinforce archive logic, fragmented identity themes, or interface tension, but they should never block usability. The website should avoid overusing animation, excessive hover tricks, or decorative transitions with no narrative or structural purpose.

CN:
交互应该服务于阅读、顺序体验和项目氛围，而不是和作品内容抢注意力。早期交互应保持稳定、清晰、低复杂度。导航必须在不依赖“试错探索”的情况下就能被理解。后续可以用动效、转场和互动细节去加强档案逻辑、碎片身份主题或界面张力，但绝不能影响基本可用性。网站应避免滥用动画、夸张 hover 技巧，或没有叙事和结构意义的装饰性转场。

## Technical Requirements / 技术需求

EN:
The website should be built with Vite, React, TypeScript, and Tailwind CSS. It should use a clean and maintainable folder structure for components, pages, layouts, data, styles, docs, and project assets. The asset structure should clearly separate original final assets, process assets, and exports. The site should remain readable across desktop, laptop, tablet, and mobile. Placeholder support is required until original materials are ready, and placeholders must remain neutral rather than pretending to be finished work. Final website-facing text should default to English, while planning and production instructions may remain in Chinese.

CN:
网站应使用 Vite、React、TypeScript 和 Tailwind CSS 构建。项目结构需要保持清晰和可维护，包含 components、pages、layouts、data、styles、docs 以及项目资产等目录。资产结构必须明确区分原创最终资产、过程资产和导出文件。网站需要在 desktop、laptop、tablet 和 mobile 上保持可读性。在原创材料尚未准备好之前，必须支持 placeholder，但 placeholder 必须保持中性，不能假装成已经完成的作品。最终面向网站观众的文案默认使用英文，而规划和制作过程中的任务说明可以继续使用中文。

## Out of Scope for Early Stage / 早期暂不做的内容

EN:
The early stage does not need heavy animation, advanced interactive experiments, large filtering systems, overly complex transitions, or fully completed final content for every project. It also does not need fake polish created by stock assets or invented case-study outcomes. The early priority is structure first, content second, visual polish third, and animation fourth.

CN:
早期阶段不需要重动画、不需要高级互动实验、不需要大型筛选系统、不需要过于复杂的转场，也不需要每个项目一开始就拥有完整的最终内容。同样，也不需要依赖 stock 资产或虚构案例结果来制造“看起来很完整”的假精致。早期优先级应当始终是：先结构，再内容，再视觉打磨，最后才是动画。

## Success Criteria / 成功标准

EN:
An early version of the website is successful if the structure is clear, the portfolio direction is recognizable, the four main projects can grow into strong case studies, the placeholders remain honest and neutral, the user knows where to place assets, and the website does not feel generic. It should also remain readable on both desktop and mobile and should already suggest a distinctive but controlled identity.

CN:
如果一个早期版本的网站满足以下条件，就可以认为它是成功的：结构清晰、作品集方向可辨认、四个主要项目都能够继续成长为完整案例、placeholder 保持诚实和中性、你清楚知道每类资产该放在哪里，而且整体网站不会显得普通模板化。同时，它还应在桌面端和移动端都保持可读，并且已经能传达出一种独特但受控的身份感。

## Future Iteration Rule / 后续迭代规则

EN:
This requirements document is expected to evolve during production. Future updates should respond to content readiness, project quality, application priorities, visual testing, and technical feasibility without abandoning the core positioning. Whenever this requirements document is updated in the future, update both the English and Chinese versions together. Do not update only one language unless I explicitly ask for a single-language draft. If a new feature, design rule, project direction, or constraint is added, it should appear in both EN and CN.

CN:
这份需求文档在制作过程中预期会持续演化。后续更新应根据内容准备情况、项目质量、申请优先级、视觉测试和技术可行性来调整，但不能偏离核心定位。后续每次更新这份需求文档时，都需要同步更新英文和中文两个版本。除非我明确要求只写单语草稿，否则不要只改其中一种语言。如果新增功能、设计规则、项目方向或限制条件，必须同时出现在 EN 和 CN 里。

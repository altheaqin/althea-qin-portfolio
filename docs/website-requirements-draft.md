# Portfolio Website Requirements Draft

## 1. Document Purpose

This document is the first working draft of the portfolio website requirements.

Its purpose is to provide a shared direction for:

- website structure
- content scope
- page logic
- portfolio positioning
- visual and interaction principles
- future iteration decisions

This draft is intentionally broad.

It should guide production without locking the final website too early.

---

## 2. Project Background

This website is being built for 2027 Fall Hong Kong master’s applications.

It is not only a container for portfolio projects.

It is also one of the portfolio projects itself.

The website should communicate a profile positioned between:

- visual communication
- interactive media
- creative technology
- brand storytelling
- commerce-oriented design
- AI-assisted workflow

The website should not feel like:

- a generic startup portfolio
- a traditional fine art gallery
- a random image dump
- a purely personal blog

Instead, it should present a clear creative identity:

> I build visual systems that turn identity, commerce, cultural memory, and interactive storytelling into applied creative experiences.

---

## 3. Core Goals

The website should serve several goals at the same time.

### 3.1 Application Goal

Support Hong Kong master’s applications by showing:

- originality
- visual literacy
- process awareness
- project thinking
- editorial clarity
- cross-disciplinary potential

### 3.2 Portfolio Goal

Present selected work as structured case studies rather than isolated images.

Each project should demonstrate not only final output, but also:

- context
- objective
- process
- decision-making
- reflection

### 3.3 Positioning Goal

The website should position the owner as someone who can connect:

- identity
- image systems
- platform thinking
- visual storytelling
- commerce
- audience understanding
- AI-assisted creative workflow

### 3.4 Project Goal

The website itself should function as proof of:

- interface thinking
- information architecture
- curation ability
- visual system design
- digital presentation skills

---

## 4. Target Audience

Primary audience:

- admissions reviewers
- program faculty
- interviewers
- potential collaborators

Secondary audience:

- creative industry recruiters
- design-related employers
- people reviewing portfolio links quickly on mobile or desktop

The website should work for viewers who may only spend a short time scanning.

Therefore, the structure must be readable quickly while still rewarding deeper exploration.

---

## 5. Website Positioning Statement

The portfolio website should be positioned as:

an editorial visual archive of original work, process, and applied creative systems.

Key positioning characteristics:

- image-forward
- process-aware
- conceptually coherent
- commercially aware
- visually distinctive
- not overdecorated

---

## 6. Content Principles

### 6.1 Originality Principle

All final portfolio content must be based on original work.

This includes:

- original photography
- original drawings
- original design work
- original process documents
- real project materials
- truthful workflow records

The website must not use:

- stock images as final portfolio content
- copied designs
- unlicensed visual material
- invented clients
- invented project outcomes
- fake case studies
- generic AI-generated final artworks as substitutes for original work

### 6.2 AI Use Principle

AI may be used as a support tool for:

- research assistance
- workflow organization
- brainstorming
- prototyping
- interface implementation
- documentation support
- coding support

However, AI should not replace the owner’s authorship in the final portfolio outcomes.

### 6.3 Curation Principle

The website should prioritize:

- fewer but stronger works
- clear project logic
- visible process
- honest scope
- careful sequencing

Weak material should not be forced into the website simply to fill space.

---

## 7. Scope of the First Website Version

The first usable version of the website should include:

- homepage
- projects index
- four main project pages
- about page
- custom 404 page

The first version does not need:

- heavy animation
- advanced interactive experiments
- large-scale filtering systems
- overly complex transitions
- fully completed final project content

The first version should focus on:

- clear architecture
- reusable page structure
- placeholder support
- asset organization
- future scalability

---

## 8. Initial Project Scope

The website will initially center on four main projects.

### 8.1 Interactive Portfolio Website

This project treats the website itself as a case study.

It should eventually show:

- sitemap
- wireframes
- visual direction
- layout evolution
- screenshots
- mobile adaptation
- navigation logic

### 8.2 404 Self Live2D Interactive Avatar

This project explores fragmented identity and interactive character presentation through original visual material.

It should eventually show:

- sketches
- redesign process
- layer breakdown
- rigging process
- expression states
- final demo

### 8.3 AI-assisted E-commerce Brand System

This project presents a real applied workflow connecting research, design, packaging, content, and AI-assisted support.

It should eventually show:

- research
- insight
- brand decisions
- packaging development
- content logic
- workflow evidence
- final outputs

### 8.4 Portrait Persona System

This project presents portrait photography as identity construction rather than simple image display.

It should eventually show:

- persona grouping
- contact sheets
- editing process
- lighting or styling logic
- final portrait selections
- presentation context

---

## 9. Functional Requirements

At the current stage, the website only needs lightweight core functions.

### 9.1 Navigation

The website should provide:

- clear homepage entry
- project index access
- direct project detail navigation
- about page access
- 404 fallback page

Navigation should feel stable and readable.

It should not depend on experimental interaction to be usable.

### 9.2 Project Presentation

Each project should have a reusable structure that can support:

1. hero
2. one-line concept
3. context
4. objective
5. role
6. process
7. final outputs
8. reflection

### 9.3 Placeholder Support

The website must support neutral placeholders before final assets are ready.

Placeholders should:

- not imitate fake finished work
- clearly reserve space for future original materials
- maintain layout stability

### 9.4 Responsive Reading

The website should remain readable on:

- desktop
- laptop
- tablet
- mobile

The mobile version should preserve content hierarchy even if layouts become more linear.

---

## 10. Visual Requirements

The website should feel editorial, image-forward, and intentional.

At this stage, visual requirements should remain directional rather than fixed.

### 10.1 Desired Qualities

- archival
- calm but distinctive
- slightly experimental
- text-legible
- process-friendly
- not template-like

### 10.2 Things to Avoid

- startup SaaS styling
- generic portfolio grids with no identity
- overly polished commercial clichés
- excessive animation
- decoration without content purpose

### 10.3 Visual Development Rule

The visual language should develop after the structure and content become clearer.

Future aesthetic directions may draw from:

- dreamlike archive
- fragmented interface
- digital memory
- quiet glitch residue
- cultural trace

But these should only be developed when supported by real content.

---

## 11. Content Structure Requirements

The website should balance two layers:

### 11.1 Fast-reading Layer

For quick reviewers, each page should communicate:

- what the project is
- why it matters
- what kind of thinking it demonstrates

### 11.2 Deep-reading Layer

For longer review, each page should allow:

- process reading
- image sequencing
- decision understanding
- project reflection

This means the website should be easy to skim, but still meaningful in detail.

---

## 12. Asset Management Requirements

The website must be built around a clean asset structure so the portfolio can grow without becoming disorganized.

Requirements:

- separate original final assets from process assets
- preserve export folders
- maintain project-based asset grouping
- allow future asset inventory tracking
- make replacement of placeholders straightforward

The current asset placement logic should continue to be used unless later iteration reveals a better structure.

---

## 13. Writing and Language Requirements

Default rule for website-facing text:

- all final website copy should be written in English by default

Working rule for production process:

- planning discussion and task instructions may remain in Chinese

This allows the production workflow to stay efficient while keeping the final portfolio internationally readable.

---

## 14. Non-functional Requirements

The website should be:

- stable
- readable
- maintainable
- easy to update
- scalable across future project additions

It should also support:

- incremental content replacement
- iterative design refinement
- future alignment with PDF portfolio structure

Performance should be reasonable, but extreme optimization is not a priority in the first version.

---

## 15. Constraints

Current known constraints:

- original materials may not all be ready at once
- project depth will vary
- some case studies will evolve over time
- visual direction should not be finalized too early
- the website should not depend on fake assets to look complete

Because of this, the website must be designed to grow gradually.

---

## 16. Success Criteria for Early Versions

An early version of the website can be considered successful if:

- the site structure is clear
- the website already reflects a recognizable portfolio direction
- the four main projects can each grow into full case studies
- placeholders are neutral and honest
- the user knows where to place assets
- the site does not feel generic
- the site remains readable on mobile and desktop

---

## 17. Future Iteration Areas

This document is expected to evolve.

Likely future additions:

- more detailed content requirements for each project
- copywriting guidelines
- image ratio and sizing rules
- homepage storytelling strategy
- project ordering logic
- motion guidelines
- PDF alignment rules
- asset inventory method
- evaluation criteria for what to keep or remove

---

## 18. Current Draft Status

This is a working draft, not a final locked specification.

It should be revised during production based on:

- content readiness
- project quality
- application priorities
- visual testing
- technical feasibility

The main goal right now is not to define everything.

The main goal is to keep the website coherent while leaving room for stronger decisions later.
